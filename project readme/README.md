# Project Readme 编写规范

本文件供 Claude Code 读取，用于在每次版本更新时生成项目文档。

## 命名规则

- 格式：`YYYY-MM-DD-HHMM.md`
- 示例：`2026-05-07-0523.md`
- 时间必须取文件生成时的北京时间（GMT+8）！
- 存放在 `project readme/` 目录下
- 读取目录中已有文件，取最新日期的作为"上一版本"参照

## 要求

- 不能修改原有的md文档，必须新建一个md文档
- 参考上一个最新的md文档，文档命名是按照日期加时间的方式，直接读取最新的一个md文档即可。你可以获取当前日期和时间后，找离当前最近的一个文档，也可以将文件夹内的文档进行排序，得到最新的一个md文档。
- 在最新的一个md文档上进行修改

## 编写流程

1. 先读取 `project readme/` 目录中日期最新的 `.md` 文件作为上一版本基准
2. 结合当前代码实际状态（读取源文件，不要凭记忆或对话推断）撰写新版本
3. 在上一版本内容基础上进行**完善和更新**，而不是从零写起
4. 新文件以当天日期命名，不修改、不覆盖已有文件

## 正文规则

- **不要**在正文中提及"本版本改了什么"、"新增"、"删除"等改动描述
- **不要**写变更日志、diff、对比上一版本的内容
- 正文就是一份**完整的、独立的当前版本项目文档**，像从来没写过旧版一样
- 保留上一版本中仍然有效的所有内容，更新已变化的部分，删除已废弃的部分
- 如果上一版本的某个章节在当前版本完全不再适用，直接删除该章节

## 必须包含的章节

1. **基本信息** — 项目类型、主体类型、AppID、云环境、基础库版本、目录结构
2. **项目结构** — 文件树（ascii tree），标注每个文件/目录的用途
3. **页面路由与导航** — 表格列出所有页面路径、是否 tab、说明
4. **云函数 API** — 表格列出所有 type、功能、参数、涉及集合
5. **数据库集合** — 每个集合的字段/类型/说明表格，以及设计要点
6. **页面功能详情** — 每个页面的 UI 描述和交互逻辑，包含关键样式参数
7. **用户强调过的要求** — 汇总用户在对话中反复纠正过的偏好和规则
8. **未来拓展规划** — 已知的后续计划（如支付、主体升级等）

## 版本变更章节（末尾）

在文档最末尾，写一个 **版本变更** 章节，格式固定为：

```
## 版本变更

**v[新版本日期] 相对于 v[上一版本日期] 的改动**：

1. **改动标题**：简短描述改了什么
2. ...
```

规则：
- 每条用一两句话说清楚改了什么，不写技术实现细节
- 按重要性排序，最重要的改动放第一条
- 只描述对外可见的功能/行为变化，不描述代码重构
- 数量控制在 3-7 条

## 需要读取的源文件（获取最新状态）

写文档前必须读取以下文件确认当前状态：

- `miniprogram/app.json` — 页面注册和 tabBar 配置
- `miniprogram/app.js` — 云开发初始化参数
- `cloudfunctions/quickstartFunctions/index.js` — 云函数全部逻辑
- `miniprogram/pages/index/index.js` 和 `.wxml` — 首页
- `miniprogram/pages/profile/profile.js` 和 `.wxml` — 个人中心
- `miniprogram/pages/show/show.js` 和 `.wxml` — 数据展示
- `miniprogram/custom-tab-bar/index.js` 和 `.wxml` — tabbar

不要凭记忆写，必须读文件确认。

## 用户偏好记录

以下是从用户反复强调中提取的规则，写文档时需体现：

- 登录必须用 `chooseAvatar` + `type="nickname"`，不用 `getUserInfo`（返回默认值）
- `getUserProfile` 已失效不可用
- `getPhoneNumber` 个人主体无权限，个体工商户后才能用
- 按钮文字居中用 flex 布局，不要依赖 line-height
- 弹窗需避开自定义 tabbar（padding-bottom + z-index）
- 数据库以 openid 为唯一标识，在云函数服务端取值
- 不要随意修改已有样式，聚焦用户真正提出的问题
- 本地调试操作真实云端数据库
- 上传云函数用云端安装依赖
